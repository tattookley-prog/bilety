# Билет №6 — Docker + MediaWiki

## Что требуется по заданию
На BR-SRV поднять стек MediaWiki + MariaDB в контейнерах (Docker Compose), обеспечить запуск сервисов и доступ к вики по HTTP.

## Теория
Контейнеризация изолирует приложение и зависимости в единый переносимый объект — контейнер.
Образ (image) — шаблон, контейнер — запущенный экземпляр образа.
Docker Compose управляет несколькими связанными сервисами через единый YAML-файл.
В стеке MediaWiki+MariaDB важно сетевое имя сервиса БД: приложение обращается к БД по имени контейнера (например, `mariadb`), а не по `localhost`.
Volumes сохраняют данные между перезапусками контейнеров (база, конфиги).
Проброс портов (`-p`) публикует порт контейнера наружу, например `8080:80`.
`LocalSettings.php` — основной конфиг MediaWiki после установки.

Для защиты полезно объяснить не только команды, но и логику потока запроса/данных:
что является точкой входа, где хранится состояние, и каким шагом подтверждается корректность.
Хорошая практика ответа: сначала назвать роль сервиса, затем ключевой конфиг, затем проверочную команду.
Так преподавателю видно, что вы понимаете причину настройки, а не только повторяете команды.

## Ключевые файлы и пути конфигурации
- `~/wiki.yml` (compose-файл)
- Том данных MariaDB (в compose)
- `LocalSettings.php` (внутри/снаружи контейнера)

## Основные команды
- `docker ps` — список запущенных контейнеров.
- `docker compose -f ~/wiki.yml up -d` — запуск стека.
- `docker logs <container>` — диагностика.
- `docker exec mariadb ...` — проверка БД из контейнера.
- `curl -I http://localhost:8080` — проверка HTTP-ответа MediaWiki.

## Как проверить результат
Используйте справочник в конце скрипта: `scripts/ticket06_docker_wiki.sh`.

## Частые вопросы преподавателя / на что обратить внимание
- Почему `localhost` не подходит как хост БД внутри контейнера wiki.
- В чём разница между образом и контейнером.
- Что удаляется при `docker compose down -v` и почему это важно.
