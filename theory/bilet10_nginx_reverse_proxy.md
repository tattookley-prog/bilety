# Билет №10 — nginx reverse proxy

## Что требуется по заданию
Настроить на HQ-RTR обратный прокси nginx для доменных имён `moodle.au-team.irpo` и `wiki.au-team.irpo` с проксированием на backend-серверы.

## Теория
Обратный прокси принимает запросы клиентов и пересылает их на внутренние backend-сервисы.
В nginx это делается директивой `proxy_pass` в нужном `server`/`location`.
`server_name` позволяет разделять виртуальные хосты по имени сайта.
Прокси-заголовки (`Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`) передают backend-у исходный контекст клиента.
Если Apache и nginx одновременно претендуют на порт 80, возникает конфликт — обычно Apache переводят на другой порт (в стенде `8081`).
Итог: клиент видит единые доменные имена, а backend-ы могут жить на разных узлах/портах.

Для защиты полезно объяснить не только команды, но и логику потока запроса/данных:
что является точкой входа, где хранится состояние, и каким шагом подтверждается корректность.
Хорошая практика ответа: сначала назвать роль сервиса, затем ключевой конфиг, затем проверочную команду.
Так преподавателю видно, что вы понимаете причину настройки, а не только повторяете команды.
Полезно отдельно объяснить, почему проверка с `Host`-заголовком точнее обычного `curl -I http://127.0.0.1/`.

## Ключевые файлы и пути конфигурации
- `/etc/nginx/nginx.conf`
- `/etc/nginx/conf.d/reverse-proxy.conf` (или `sites-available`/`sites-enabled`)
- Конфиги backend-сервисов (Apache/Moodle, MediaWiki)

## Основные команды
- `nginx -t` — проверка синтаксиса.
- `systemctl is-active nginx` — статус службы.
- `ss -tlnp | grep ':80'` — кто слушает порт 80.
- `curl -I -H 'Host: ...' http://127.0.0.1/` — проверка маршрутизации по host header.

## Как проверить результат
Используйте справочник в конце скрипта: `scripts/ticket10_nginx_proxy.sh`.
